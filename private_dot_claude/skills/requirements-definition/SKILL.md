# 要件定義 ルーター（見分けるスキル）

あなた（Claude/Codex）が**実装に飛びつく前に**、要求を獲得・構造化・合意・検証する
ための入口。**規模を見極め（ティア判定）**、短いインテークを行い、適切なサブスキルへ
ルーティングし、実装してよいかの**ゲート**を通す。

出典: IPA「ユーザのための要件定義ガイド 第2版」を AI コーディング文脈に翻案。
サブスキル（requirements-business / -system / -verify / -documents）は**この入口から呼ぶ**。
これらは**独立した兄弟スキル**で、Skill ツールに**スキル名**を渡して起動する（ファイルパスを
推測しない／このスキルの配下にネストしていない）。

---

## 0. 原則（これだけは外さない）

1. **手段（機能）より先に、目的を確かめる。** 会話が「〇〇機能を作って」から始まったら、
   目的・課題への問い返しを最優先する。これがトリガーの第一指標。
2. **「現行踏襲」「今と同じで」は要件ではない。** 可視化・分析しないまま受け入れない。
3. **要求 ≠ 要件。** 要求を文書化・仕様化し**合意**したものが要件。合意なきものを実装前提にしない。
4. **数値化されない非機能要求は要件ではない。**「速く」「使いやすく」は数値に落とす。
5. AI が価値を出すのは**構造化・整合チェック・トレーサビリティ・抜け漏れ検出**。
   合意形成・経営判断・契約は人間に委ねる（提示・支援までがAIの役割）。

> ここでの「目的」は経営目的に限らない。**プロダクト目的・利用者価値・運用改善・開発体験**も
> 同格の目的。OSS/個人開発/社内小ツールでは「経営ゴール」の語にこだわらず目的を確かめる。

## 1. 起動トリガー（このスキルを使うべきシグナル）

ユーザの発言・状況に次が現れたら、いきなり実装せず本スキルでインテークする：

- **目的不在で実装へ**:「とりあえず作って」「まず動くものを」「〇〇機能を追加して」（なぜ／何のためが無い）
- **現行踏襲**:「今と同じで」「現行どおり」「既存を再構築」（現状の可視化なし）
- **偽の確定**:「要件は固まってる」だが内容が曖昧／口頭のみ／数値が無い
- **非機能の先送り**:「非機能は後で」「まず機能だけ」（性能・セキュリティ未検討）
- **スコープ膨張**:「あれもこれも」「全部やって」（優先順位なし）
- **合意の欠落**:「一部の人しか知らない」「経営層は関わってない」「SH合意は未」
- **工程短縮の正当化**:「急いでるので要件定義は省略／短縮したい」
- **未決のまま完了宣言**:「要件定義は完了」だが未決事項・TBDが残る

> 入口のトリガー判定は**このルーターが一手に担う**。サブスキルは「工程の中身」だけを持つ。

## 2. 要件定義の全体地図（1枚）

```
[目的/企画]→ BR ビジネス要求定義 ─→ SR システム化要求定義 ─→[設計・実装]
              (要求=ニーズ)            (要件=合意された仕様)
   獲得→分析→文書化            仕様化→検証→妥当性確認
        └──── 横断: スコープ/トレーサビリティ/未決/完了判断（requirements-verify）────┘
                    成果物: 要求一覧・非機能要件書・各種一覧・CRUD図 …（requirements-documents）
```

- **BR ビジネス要求定義**: 現状把握→問題・課題→ゴール→手段→体系化→優先順位→交渉→文書化
- **SR システム化要求定義**: 要求を機能/非機能の「要件」へ仕様化
- **検証(Verification)**: 要件が要求を抜け漏れ・矛盾なく満たすか（作り方が正しいか）
- **妥当性確認(Validation)**: その要求が目的を実現できるか（正しいものを作っているか）

## 3. 用語（プロジェクト開始時に揃える）

| 用語 | 定義（本ガイド/JIS X 0166 準拠） |
|---|---|
| 要求 (requirement) | ニーズと制約を変換・表現した文。まだ合意前。 |
| 要件 | 要求を文書化・仕様化し**合意**したもの。次工程の入力。第三者に伝達可能な形式。 |
| ビジネス要求 | 利害関係者要求(StRS)。なぜ作るか・業務でどう使うか。 |
| システム化要求 | システム要求仕様(SyRS)。機能・性能・**制約**・属性の構造化集合。 |
| 検証 (Verification) | 記述された要件が要求を抜け漏れ・誤りなく満たすか。 |
| 妥当性確認 (Validation) | 定義された要求自体が目的を実現でき、手段として妥当か。 |

## 4. 規模ティア判定（最初にやる ── 過剰適用を防ぐ）

**どれだけ重く回すかを最初に決める。** 小さい依頼に重い要件定義フルセットを課さない。

| ティア | 条件（おおむね） | やること |
|---|---|---|
| **Light（軽量パス）** | 既存システムへの**単一機能追加/小バグ/UI文言**。目的が自明、影響1〜2ファイル、非機能影響なし、受入条件が言える | **4点だけ確認**して実装可（後述）。BR/SRフルはスキップ |
| **Standard（標準）** | 新機能群／複数SH／非機能影響あり／データ構造変更 | router→business→system→verify を**目的に必要な分だけ**回す |
| **Deep（重量）** | 新規開発／再構築／基幹業務／多部門 | BR→SR→verify＋成果物セットをフルに回す |

### Light（軽量パス）の最小4点
1. **目的**: この変更で何を解決するか（1行）
2. **受入条件**: 「できた」と言える観測可能な条件（数件）
3. **対象外**: やらないこと（スコープ境界）
4. **影響・非機能**: 触る範囲、性能/セキュリティ/互換性への影響の有無

→ 4点が言えれば **critical ゲートを満たしたとみなして実装着手可**。
成果物は requirements-documents の「小規模ミニセット」（ミニ要求一覧＋受入条件＋対象外）で足りる。

**「非機能影響あり」＝ Standard へ格上げの目安**: 外部サービス/新規依存の追加・データスキーマ変更・
認証認可に触れる・性能/可用性/セキュリティ要件が絡む・失敗時の挙動が非自明、のいずれか。
受入条件が割れた瞬間も Standard へ。

**判定手順**: 目安に**明らかに**当てはまる（例: 決済・認証の新規導入）なら、最初から Standard と宣言する。
当てはまるか**回答次第で決まる**（例: メール通知＝既存SES流用なら新規依存でない／失敗時ログのみなら非自明でない）
ときは、**まず Light の4点を聞き、回答を見てティアを確定**してよい。
Light 確認の結果「新規依存あり／失敗時が非自明」と分かったら Standard へ格上げ。

> 迷ったら Standard。Light は「明らかに小さい」ときだけ。
> 目的が言えない（なぜ作るかが不明）なら、Light でも**目的の1問だけは先に**確認する。

## 5. ルーティング表（Standard/Deep の現在地 → 使うスキル）

| いまの状況 | 使うスキル |
|---|---|
| 目的・課題・ゴールが曖昧／手段から会話が始まった／現状把握が必要 | **requirements-business**（獲得・分析） |
| 要求は出たが優先順位・合意が未／スコープが膨らむ | **requirements-business**（優先順位・交渉） |
| 要求は固まった→機能/非機能の仕様に落としたい | **requirements-system**（仕様化） |
| 仕様の抜け漏れ・矛盾・整合・目的適合を点検したい／完了判断したい | **requirements-verify** |
| 要求一覧・非機能要件書・各種一覧・CRUD図など成果物を出力したい | **requirements-documents** |

- 多くの新規依頼は **requirements-business から開始**し、SR → verify と下流に進む。
- **逆流ルール**（verify等で問題が出たら戻る先）:
  - 目的ズレ・採否の問題 → **business** へ
  - 仕様の不足・曖昧 → **system** へ
  - テンプレ/成果物の形 → **documents** へ
  - 完了判定・トレース・未決 → **verify** に残す
- スキルを使うときは Skill ツールで該当スキル名を呼ぶ。

## 6. インテーク（Standard/Deep の入口で必ず聞く3問）

トリガー検知後、まずこの**3問**を具体的に問う（軽くてよい。回答を要求一覧の起点にする）:

1. **なぜ／何のために** — この開発で解決したい問題・目的は？（業務/プロダクト/利用者価値）
2. **誰のために／誰が使う** — 主な利用者・関係者は？ 合意が必要なのは誰か？
3. **成功とは** — 「うまくいった」と言える状態・受入条件・制約（予算/期日/技術/既存資産/法規）は？

> この3問は requirements-business の獲得（BR.1.2〜1.3）の入口でもある。
> 制約（技術スタック・既存DB・コンプラ・締切）は3問目で必ず拾う＝AI実装の可否を左右する一級情報。

## 7. 実装前ゲート（"GO/NO-GO" 判定）

実装（コード生成・設計確定）に進む前に確認する。**判定は2段階**:

**critical（1〜3）— 1つでも✕なら実装不可。**
1. **[critical] 目的紐付け**: 実装対象が目的（業務/プロダクト/利用者価値）に紐づくか。
   ✕→ requirements-business（BR.1.3 ゴール抽出）
2. **[critical] 合意**: 主要ステークホルダが特定され、要求に合意しているか。
   ✕→ requirements-business（BR.0 SH特定・BR.2.4 交渉）
   - **単独依頼者モード**: 依頼者が唯一のSH／PO代理なら、**依頼者の確認をもって合意**とみなし、
     その前提（誰の合意か）を明記して進む。「合意できないから常にNO-GO」にしない。
3. **[critical] 要件化**: 要求が抜け漏れ・曖昧なく要件化されているか
   （＝requirements-verify 基準で **未カバー要求0 かつ 孤立機能0**）。
   ✕→ requirements-system（SR.1 仕様化）→ requirements-verify

**non-critical（4〜5）— ✕ならリスクを明示。軽微/小規模なら条件付き着手可、大規模なら不可。**
4. **検証・妥当性確認**: 整合・網羅（検証）と目的適合（妥当性確認）が点検されたか。✕→ requirements-verify
5. **未決事項**: 残った未決は台帳化され、影響が許容範囲でオーナー合意があるか。✕→ requirements-verify（未決・完了判断）

> critical（1〜3）が全て○で「実装着手可」。○×と根拠をユーザに明示する。
> Light ティアでは §4 の4点確認が critical の充足に相当する。
> **提示タイミング**: 第一応答ではゲートを長々と並べない。インテーク（3問/4点）を先に行い、
> 回答が揃った時点で GO/NO-GO を○×と根拠で示す。第一応答では「揃ったら実装に進む」と一言予告すれば足りる。

---

## 8. 最小ワークフロー

1. **ティア判定**（§4）: Light か Standard/Deep か。
2. **インテーク**（§6）: Standard/Deep なら3問。Light なら4点。
3. **見分け**（§5）: ルーティング表で現在地を判定し、該当サブスキルを起動。
4. **成果物**: 節目で requirements-documents から要求一覧などを出力し合意の土台にする。
5. **ゲート**（§7）: 実装に進む前に通す。critical NG なら戻る。
6. **トレース**: 要求にIDを振り、後工程（設計・実装・テスト）と紐づけ続ける（verify参照）。

> 規模が小さいほど軽く回す。critical ゲート（目的・合意・要件化）は規模に関わらず通すが、
> Light では「4点確認」がその最小形。過不足なくテーラリングする。
