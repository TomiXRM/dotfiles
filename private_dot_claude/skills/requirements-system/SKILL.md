# システム化要求の仕様化（SR.1）

BR で合意した「要求（ニーズ）」を、第三者に伝達でき実装者が作れる「要件（仕様）」へ変換する。
**機能要件と非機能要件の両面**で曖昧性を排除し、定量化する。

前提: `requirements-business` で要求一覧（合意済み）がある。
仕様の点検（検証・妥当性確認）は `requirements-verify`。出力形式は `requirements-documents`。

---

## 0. 仕様化キックオフ（最初の問い）

> BR成果物（要求一覧・業務フロー）を入力に、次の観点で仕様化の漏れを洗う:
> (a) 自然言語で書かれた機能ルールのうち、デシジョンテーブル/状態遷移図に置き換えるべきものは？
> (b) 非機能要件で数値化されていない形容詞・副詞は？
> (c) ドキュメント間で対応が取れていない項目は？

**成果物間の関連を先に定義する**（プロセス系/データ系/インターフェース系/非機能系）。
不整合の根源はドキュメント間関係の無理解。先に関連マトリクスを引く。

**業務用語定義を先行**させる（業界/企業固有語）。後で解釈齟齬が連鎖する。
→ requirements-documents の業務用語定義(DD.7.1)。

---

## 1. 機能要件の仕様化

ビジネスルールは**種別ごとに記法を選ぶ**。自然言語フリーフォーマットは抜け漏れ・曖昧性の温床。

| ルール種別 | 例 | 推奨記法 |
|---|---|---|
| 事実・構造 | エンティティと関連 | ER図（mermaid） |
| 契機・順序 | 業務の流れ | 業務フロー / BPMN（mermaid flowchart） |
| 制約・条件分岐 | 条件×結果の組合せ | **デシジョンテーブル** |
| 状態・ライフサイクル | 受注→出荷→請求… | **状態遷移図**（mermaid stateDiagram） |
| 計算・推論 | 料金計算式 | 計算式＋具体値の例 |

- **AIアクション**:「以下のルール記述を分類（事実/契機/制約/推論/計算）し、最適記法を推薦して
  書き直す」。条件組合せは必ずデシジョンテーブルで網羅（漏れを可視化）。
- 振る舞いは**シナリオ（基本・代替・例外）＋事前条件・事後条件**で書く
  → システム化要求仕様(DD.9.1)。
- 例外・特殊処理を必ず抽出（正常系だけにしない）。これが品質に直結。

---

## 2. 非機能要件の仕様化（定量化必須）

「高速」「大量」「十分な」は不可。**等号・不等号で数値化**する。形容詞・副詞を使ったら必ず数値補足。
IPA非機能要求グレード／ISO/IEC 25010 の観点で網羅する。

| 領域 | 代表項目（数値で書く） |
|---|---|
| 可用性 | 運用時間・稼働率(99.9%等)・RTO・RPO・計画停止 |
| 性能・拡張性 | 画面レスポンス(◯秒以内)・同時接続数・ピーク処理量・データ量・拡張余地 |
| 運用・保守性 | 運用スケジュール・バックアップ・障害対応体制 |
| 移行性 | 移行方式・対象データ量・並行稼働期間 |
| セキュリティ | 認証認可・暗号化・ログ・不正アクセス対策 |
| システム環境 | HW/OS/MW/ネットワーク |
| **制約・前提（与件）** | 技術スタック・既存DB/資産・予算・期日・法規/コンプラ・互換性・データ移行有無 |

> **制約・前提は AI 実装の可否を左右する一級情報。** 機能/非機能と別に必ず明示する。

- **コスト影響大の項目（24時間無停止・DR・高可用）は経営層と合意**。後回しは設計手戻り確定。
- **AIアクション**:「非機能要件書を受け取り、形容詞・副詞を含む記述を全列挙→数値基準候補
  （レスポンス◯秒、可用性◯%）を提示」。→ **非機能要件書**(DD.12)。

---

## 3. 要件の書き方（曖昧性の排除）

- **一文一義・短文主義**: 1文 平均40字・最大200字。接続助詞「が」を多用しない。
  「は」「の」「を」の連続を避ける。口頭言語と書記言語を混在させない。
- **主語を明示**。複数解釈できる語・定義のない抽象語を残さない。
- **AIアクション（曖昧性スキャン｜draft時の自己チェック）**:「要件文書を受け取り、(a)主語不明
  (b)複数解釈可 (c)定義なし抽象語 (d)接続助詞『が』多用、を全指摘し改善案を出す」。
  ※これは**書きながらの自己チェック**。完成物に対する第三者点検（作成者を除く）は requirements-verify。

## 5. 実装準備ブリーフ（Implementation Readiness Brief）

要件をAI/開発者が**実装に着手できる形**に束ねる、SRの締め成果物。Light ティアでもこれだけは出す。
```markdown
## 実装準備ブリーフ: <機能名>
- 目的: <なぜ作るか 1行 / 紐付くゴール>
- 対象 / 対象外: <やること / やらないこと（スコープ境界）>
- 受入条件: <観測可能な合否条件（箇条書き）>
- 影響範囲: <変更する画面/モジュール/データ・既存互換性・データ移行有無>
- 非機能影響: <性能/可用性/セキュリティへの影響と数値基準>
- 制約・前提: <技術スタック/既存資産/期日/法規>
- テスト観点: <正常/例外/特殊で確認する点>
- ロールバック/リリース方針: <失敗時の戻し方・段階リリース>
- トレース: <BR-ID / SR-ID>
```
→ requirements-documents にもテンプレあり。verify はこのブリーフを点検対象にする。

## SR.1 の本質（3行）
1. 要求（ニーズ）を、第三者に伝達でき実装者が作れる**要件（仕様）**に変換する工程。
2. **機能は記法選択で網羅性を、非機能は数値化で一意性**を担保する。自然言語フリーは禁。
3. 業務用語定義と成果物間関連を**先に**固めると、後段の不整合・手戻りが激減する。

完了の目安: 機能（シナリオ/記法）と非機能（数値）が要件として記述され、各種一覧で網羅。
→ 次は `requirements-verify`（検証・妥当性確認・トレーサビリティ・完了判断）。
